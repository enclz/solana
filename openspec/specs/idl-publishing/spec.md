## Requirements
### Requirement: SDK package layout

The repo SHALL contain a `sdk/` directory with its own `package.json` named `@enclz/sdk`, a `tsconfig.json`, a `README.md`, and a `src/index.ts`. Generated artifacts (`sdk/src/enclz.ts`, `sdk/src/enclz.json`, `sdk/dist/`, `sdk/node_modules/`) SHALL be gitignored. The root `package.json` SHALL remain `"private": true`; only `sdk/package.json` SHALL be publishable.

#### Scenario: Fresh checkout
- **WHEN** a developer clones the repo and runs `ls sdk/`
- **THEN** they see `package.json`, `tsconfig.json`, `README.md`, `src/index.ts` and **not** `dist/`, `src/enclz.ts`, `src/enclz.json`, or `node_modules/`

#### Scenario: Root remains private
- **WHEN** any reader inspects the root `package.json`
- **THEN** the `"private": true` field is present and there is no `"files"` allowlist or publish-related metadata

### Requirement: SDK build pipeline

The repo SHALL include `scripts/build-sdk.mjs`, invoked via `npm run build:sdk`, that:
1. Runs `anchor build` if `target/idl/enclz.json` or `target/types/enclz.ts` is missing.
2. Copies `target/idl/enclz.json` → `sdk/src/enclz.json` and `target/types/enclz.ts` → `sdk/src/enclz.ts`.
3. Reads `metadata.version` from `sdk/src/enclz.json` and rewrites `sdk/package.json` `version` if it differs.
4. Runs `tsc` in `sdk/`, emitting `sdk/dist/{index.js, index.d.ts, enclz.js, enclz.d.ts, enclz.json}`.

#### Scenario: Clean build
- **WHEN** developer runs `npm run build:sdk` on a fresh checkout with `anchor` and `cargo-build-sbf` available
- **THEN** `sdk/dist/index.js`, `sdk/dist/index.d.ts`, and `sdk/dist/enclz.json` exist and `sdk/package.json` `version` matches `target/idl/enclz.json` `metadata.version`

#### Scenario: Build is idempotent
- **WHEN** developer runs `npm run build:sdk` twice in a row without source changes
- **THEN** the second run completes without error and produces byte-identical `sdk/dist/` contents

### Requirement: SDK exports

The published `@enclz/sdk` package SHALL export:
- `IDL` — the Anchor IDL JSON object loaded from `enclz.json`
- `PROGRAM_ID` — the string value of `IDL.address`
- `Enclz` — the TS type generated by `anchor build` (re-exported via `export type`)

The package SHALL declare `@coral-xyz/anchor` as a `peerDependency` (range matching root `package.json`) and SHALL NOT declare it as a `dependency`.

#### Scenario: Consumer constructs typed Program
- **WHEN** a consumer runs `npm install @enclz/sdk @coral-xyz/anchor @solana/web3.js` and writes `import { IDL, type Enclz } from "@enclz/sdk"; const program = new Program<Enclz>(IDL, provider);`
- **THEN** TypeScript compiles without errors and `program.programId.toBase58()` equals the value of `PROGRAM_ID`

#### Scenario: Anchor not pulled transitively
- **WHEN** a consumer inspects `node_modules/@enclz/sdk/package.json`
- **THEN** `@coral-xyz/anchor` appears under `peerDependencies` only, not under `dependencies`

### Requirement: Version single-sourced from program

The `version` field of `sdk/package.json` SHALL equal `metadata.version` of `target/idl/enclz.json` after every `npm run build:sdk`. The IDL `metadata.version` is itself set by `anchor build` from `programs/enclz/Cargo.toml`, so the chain is `Cargo.toml` → IDL → SDK.

#### Scenario: Cargo bump propagates
- **WHEN** developer increments `version` in `programs/enclz/Cargo.toml`, runs `anchor build`, then runs `npm run build:sdk`
- **THEN** `sdk/package.json` `version` reflects the new value without manual edits

#### Scenario: Manual SDK version edit is overwritten
- **WHEN** developer manually edits `sdk/package.json` `version` to a value different from IDL `metadata.version` and then runs `npm run build:sdk`
- **THEN** the build script restores `sdk/package.json` `version` to match the IDL

### Requirement: On-chain IDL publication scripts

The root `package.json` SHALL include four scripts wrapping `anchor idl init` / `anchor idl upgrade` for devnet and mainnet:
- `idl:init:devnet`, `idl:upgrade:devnet` — use `${QUICKNODE_DEVNET_RPC_URL}`
- `idl:init:mainnet`, `idl:upgrade:mainnet` — use `${MAINNET_RPC_URL}`

All four SHALL pass `--filepath target/idl/enclz.json` and the program ID `67i3uY4gZaidynKa8XbNW569qACSVCebwKnLpNYVtWjj` (matching `Anchor.toml`). All four SHALL be invoked via `dotenv-cli` so `${VAR}` references resolve, matching the existing `deploy:devnet` / `deploy:mainnet` pattern.

#### Scenario: Initial upload after first deploy
- **WHEN** the upgrade authority runs `npm run idl:init:devnet` once after the first program deploy on devnet
- **THEN** the IDL account is created on devnet and `anchor idl fetch --provider.cluster ${QUICKNODE_DEVNET_RPC_URL} 67i3uY4gZaidynKa8XbNW569qACSVCebwKnLpNYVtWjj` returns the same JSON as `target/idl/enclz.json`

#### Scenario: Upgrade after program change
- **WHEN** the upgrade authority deploys a new program version and runs `npm run idl:upgrade:devnet`
- **THEN** `anchor idl fetch` returns the new IDL and the IDL account's `metadata.version` matches `programs/enclz/Cargo.toml`

#### Scenario: Without upgrade authority
- **WHEN** a non-authority key runs `npm run idl:upgrade:devnet`
- **THEN** the script exits non-zero with a signature-verification error from `anchor idl`

### Requirement: Public positioning relative to "no SDK required"

The SDK README and the `docs/` submodule SHALL frame `@enclz/sdk` as program-level bindings for direct on-chain callers, distinct from the agent integration path covered by the Agent REST API + `SKILL.md` + MCP server. Specifically:
1. `sdk/README.md` SHALL include in its first or second paragraph a sentence pointing readers who are integrating an AI agent to the Agent REST API + MCP server instead.
2. `docs/REQUIREMENTS.md` SHALL include a "Program Integration Resources" section that names `@enclz/sdk` and the on-chain IDL channel, sibling to the existing "Agent Integration Resources" section.
3. `docs/MARKETING.md` SHALL include a footnote on the "No SDK required" competitive comparison clarifying that the claim refers to agent integration and that direct program integrators may use `@enclz/sdk`.

#### Scenario: README positions correctly
- **WHEN** a reader opens `sdk/README.md`
- **THEN** within the first two paragraphs they find a reference to the Agent REST API + MCP server as the recommended path for AI agents

#### Scenario: docs make asymmetry explicit
- **WHEN** a reader opens `docs/REQUIREMENTS.md`
- **THEN** they find a "Program Integration Resources" section (or equivalent heading) referencing `@enclz/sdk` alongside the existing "Agent Integration Resources" section

#### Scenario: Marketing claim retains scope
- **WHEN** a reader opens `docs/MARKETING.md` and reads the row "No SDK required" in the competitive table
- **THEN** a footnote or adjacent sentence clarifies that the claim refers to agent integration, with `@enclz/sdk` named as the program-level alternative
